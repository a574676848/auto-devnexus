# Open Source Docs Skill

## 简介

`open-source-docs` 是一个面向开源项目文档建设的 Skill。

它的目标不是“帮忙补几段文字”，而是帮助 AI 助手在接到文档任务时，优先产出更接近优秀开源项目的正式文档体系，包括：
- 项目门面型 README
- 可导航的 docs 文档中心
- 与当前产物一致的 examples 说明
- 中文友好但技术准确的表达
- 完整接入图片、许可证、致谢与贡献入口

如果你遇到下面这些问题，这个 Skill 就很适合：
- README 看起来像迭代记录，不像正式开源项目首页
- `docs/` 里文档很多，但没有清晰结构
- 图片目录里有图，正文里却没接进去
- examples 已更新，但文档还是旧说明
- 文件名和正文内容已经漂移
- 文档大多是英文堆砌，团队读起来不顺

## 这个 Skill 会帮助 AI 做什么

1. 先盘点仓库现状，识别 README、docs、examples、图片和命名漂移问题。
2. 建立适合当前项目的信息架构，而不是直接改段落。
3. 把根 README 重写为开源项目门面。
4. 把 `docs/` 整理成更稳定的文档体系。
5. 把图片、许可证、致谢、贡献说明等容易丢失的资产接入正确位置。
6. 对全文档做中文友好化，同时保留必要的技术英文。
7. 清理旧命令、旧术语、旧文案残留。

## 推荐触发场景

可以直接对安装了该 Skill 的 AI 助手这样说：

- “把这个项目的 README 改成像优秀开源项目。”
- “整理一下 docs 结构，别写成迭代说明。”
- “把图片都接进文档里，并做中文友好化。”
- “examples 已经更新了，帮我同步所有文档。”
- “文档命名和正文内容不一致，统一纠偏。”

## 使用建议

### 1. 让 AI 先盘点，再动手
如果仓库里已有不错的文档内容，最重要的是保留高质量资产并纠偏，而不是粗暴重写。

### 2. 把 README 和 docs 分开治理
README 负责对外门面，docs 负责深入说明。不要让 README 承担全部信息。

### 3. 明确告诉 AI 项目是否面向中文团队
这个 Skill 默认偏中文友好，但如果你的项目主要面向国际社区，可以让 AI 在保留结构的同时降低中文比例。

## 交付预期

使用这个 Skill 后，AI 的输出应更接近这些标准：
- README 能让陌生人 1 到 2 分钟理解项目是什么
- docs 有清晰入口和主题边界
- 图片、示例、命令、路径都与当前项目一致
- 保留许可证、致谢、贡献入口等开源项目信号
- 不再大面积出现“本次迭代”“本轮开发”这类时态化表述
